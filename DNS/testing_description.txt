ОПИСАНИЕ ТЕСТИРОВАНИЯ СИСТЕМЫ

Для обеспечения стабильной и качественной работы платформы магазина электроники DNS необходимо провести тестирование всех ее модулей. Тестирование позволит выявить проблемные участки кода, требующие доработки, и обеспечить надежную работу системы. В данном случае применяется методика тестирования "белого ящика".

План тестирования для модуля пользователя

Настольное приложение для пользователя на ПК ОС Windows процессор AMD.

Тестируемый функционал:
    Авторизация пользователя
    Регистрация пользователя
    Просмотр каталога товаров
    Поиск товаров по различным критериям
    Просмотр деталей товара (описание, характеристики, цена)
    Добавление товара в корзину
    Оформление заказа
    Просмотр истории заказов
    Просмотр личной информации пользователя (профиль)
    Управление корзиной покупок

Тип тестирования:
    Исследовательское тестирование
    Функциональное тестирование
    Нефункциональное тестирование (юзабилити, производительность)

Анализ результатов:
Благодаря применению метода тестирования "белого ящика" обеспечивается возможность обнаружения и исправления дефектов, что способствует значительному повышению качества программного продукта.

Критерии начала тестирования:
    Готовность базы данных (наполнение тестовыми данными).
    Готовность документации к программному продукту (руководство пользователя).
    Развертывание системы магазина.
    Тестовая платформа готова, вся необходимая документация в наличии.

Критерии окончания тестирования:
    Интерфейс пользователя полностью на русском языке.
    Отсутствуют критические ошибки (непопадки, блокирующие основные функции).
    Основные функции системы работоспособны в соответствии со спецификацией.
    Протестированы все необходимые сценарии для пользователя.

Нефункциональное тестирование

Предпосылки:
    Средняя производительность: предположим, что приложение развернуто на сервере среднего класса с достаточными, но не максимальными ресурсами. Это означает, что оно может обрабатывать умеренную нагрузку, но не экстремальные пики.

    База данных SQL Server: настроена на стандартном плане, предоставляя достаточную производительность для средних нагрузок.

Примерные параметры:
    Количество пользователей: 100 активных пользователей в день
    Одновременные сессии: В среднем 50-100 одновременных сессий в течение дня
    Количество товаров: 1000 товаров в каталоге
    Средний размер каталога: около 5-10 категорий товаров
    Количество заказов: от 10 до 20 в день
    Среднее время работы пользователя: 10-15 минут за сессию

Тестирование нефункциональных требований (платформа с SQL Server, средняя производительность)

1. Тестирование производительности
Цель: оценить производительность приложения и взаимодействие с SQL Server при средней нагрузке, определенной выше.

Методика:
    Нагрузочное тестирование: Имитация 75 одновременных пользователей, выполняющих основные действия (просмотр каталога, добавление в корзину, оформление заказа).
    
    Стресс-тестирование: Проверка поведения при 100 одновременных пользователях (пиковая нагрузка).
    
    Тестирование скорости: Измерение времени отклика на различные действия.

Система тестов (с конкретными данными):
Нагрузочные тесты:
    Моделирование 75 одновременных сеансов, где пользователи просматривают каталоги товаров (50%), добавляют товары в корзину (20%), оформляют заказы (30%).
    Проверка времени загрузки страниц каталога и товаров при такой нагрузке.
    Имитация 10 одновременных запросов на оформление заказов.

Стресс-тесты:
    Моделирование 100 одновременных сеансов.
    Проверка времени отклика при перегрузке системы.
    Проверка поведения SQL Server в таких условиях.

Тесты скорости:
    Время загрузки каталога (ожидание: 2-3 секунды).
    Время загрузки страницы товара (ожидание: 1-2 секунды).
    Время отклика на запрос записи на курс (ожидание: 1-3 секунды).
    Время загрузки данных пользователя (ожидание: 1-2 секунды).
    Время отклика на поиск товаров (ожидание: 1-2 секунды).

Результаты тестирования (примерные):
    Время отклика SQL Server: Среднее время отклика для запросов к SQL Server должно составлять менее 100-200 мс при нормальной нагрузке и не превышать 500 мс при пиковой.

    Время загрузки: Среднее время загрузки страниц каталога и товаров должно быть 2 секунды при обычной нагрузке.

    Пропускная способность: Приложение должно справляться с 75 одновременными пользователями, делающими до 500 запросов в минуту.

    Использование ресурсов: Загрузка процессора сервера SQL Server не должна превышать 60% при нормальной нагрузке, а нагрузка на ЦП приложения 50%.

    Ошибки и сбои: не должно быть критических ошибок и сбоев при нагрузочном и стресс-тестировании.

2. Тестирование надежности

Цель: Проверка устойчивости приложения и базы данных SQL Server в условиях среднего трафика.

Методика:
    Тестирование восстановления SQL Server: Моделирование перебоев соединения с SQL Server и восстановления после них.
    
    Тестирование отказоустойчивости SQL Server: Проверка поведения приложения при временных сбоях SQL Server.

Система тестов (с конкретными данными):
    Тесты восстановления SQL Server:
        Моделирование 10-секундного перебоя связи с SQL Server
        Проверка сохранения данных пользователя и сессии
        Проверка правильности обработки ошибок при сбое связи
    
    Тесты отказоустойчивости SQL Server:
        Эмуляция временной (5-10 секунд) задержки в ответах SQL Server
        Проверка, что критические функции (например, авторизация) остаются работоспособными
        Проверка корректности отображения данных, даже при нестабильном соединении с SQL Server

Результаты тестирования:
    Время восстановления SQL Server: Время восстановления соединения должно составлять менее 15 секунд.
    
    Количество сбоев SQL Server: ожидается не более 2-3 сбоев за 1 час работы при средней нагрузке.
    
    Сохранность данных: все пользовательские данные должны быть сохранены после сбоев связи с SQL Server.
    
    Отчет о надежности: Приложение должно оставаться работоспособным и не выдавать ошибок в 99% случаев при средней нагрузке.

Нефункциональное тестирование играет ключевую роль в обеспечении того, что система соответствует требованиям пользователей. Кроме того, нефункциональное тестирование помогает определить, насколько хорошо система справляется с реальными условиями эксплуатации.
